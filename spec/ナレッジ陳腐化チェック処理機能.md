# ナレッジ陳腐化チェック処理機能

## 1. 機能概要
ナレッジの風化・陳腐化を防ぐため、定期的なスケジュール実行によりナレッジファイルの投稿日をチェックします。一定日数が経過したファイルを順次 `check/` ディレクトリ（確認対象）、`invalid/` ディレクトリ（無効化対象）へ移動させ、Slack通知およびグラフデータベース（Neo4j）からのデータ削除を自動で行う機能です。

## 2. 実行トリガー
- **イベント**: `schedule` または `workflow_dispatch`（手動実行）
- **スケジュール**: 週次実行（UTC 日曜 20:00 = JST 月曜 05:00）
- **実行環境**: GitHub Actions (`.github/workflows/check-knowledge.yml`)

## 3. 設定値 (環境変数 / Inputs)
- `KNOWLEDGE_CHECK_DAYS`: `knowledge/` から `check/` へ移動させる経過日数（デフォルト: 30日）
- `KNOWLEDGE_INVALID_DAYS`: 投稿日から起算して `invalid/` へ移動させる経過日数（デフォルト: 90日）

## 4. 処理フロー (`scripts/check_knowledge.py`)

処理は以下の4つのフェーズで進行します。

### Phase A: `check/` → `invalid/` への移動
- `check/*.md` を走査し、Frontmatterの `date` から `KNOWLEDGE_INVALID_DAYS` 日以上経過しているものを `invalid/` に移動させます。
- 移動されたファイルの投稿者に対して、Slackで「無効化」通知を送信します。

### Phase B: `knowledge/` → `check/` または `invalid/` への移動
- `knowledge/*.md` を走査します。
- **直接無効化**: `date` から `KNOWLEDGE_INVALID_DAYS` 日以上経過しているファイルは、`check/` をスキップして直接 `invalid/` へ移動します（Slack「無効化」通知送信）。
- **確認対象へ移動**: `date` から `KNOWLEDGE_CHECK_DAYS` 日以上経過しているファイルは `check/` へ移動します。
- `check/` に移動したファイルの投稿者に対して、Slackで「再確認依頼」通知を送信します。

### Phase C: Slack通知
- Slack Webhook を利用して、Phase A / B で特定された各ファイルの投稿者にメンション付きで通知を送ります（`AUTHOR_SLACK_MAP` でGitHub名からSlack IDを解決）。

### Phase D: `invalid/` 後処理 と Neo4j 削除
- `invalid/` に移動したファイルが存在する場合、最新の無効化ナレッジ一覧を `invalid/list/YYYY-MM-DD-invalid-list.md` として生成します（古いリストは削除）。
- 無効化された各ファイルに対応する **Neo4j の Knowledge ノードおよび関連するすべてのエッジ (`WRITTEN_BY`, `USES_TOOL`, `CONTAINS_ENTITY`, `RELATED_TO`) を `DETACH DELETE` により削除**します。
  ※ Entity / Tool / Author ノード自体は他のナレッジで利用される可能性があるため残存します。

## 5. 出力と後処理
- ディレクトリ内のMarkdownファイルの移動。
- `invalid/list/` の更新。
- 変更があった場合、GitHub Actions 実行ユーザーとしてコミットを行い、リポジトリにPushします。
